# 🎩 Sorting Hat Skill —— 为你量身定制 AI 协作伙伴

> 让 AI 不再只是通用助手，而是像那个与你性格、节奏和工作方式完美契合的人一样，与你达成高效、自然、长期的协作。

Sorting Hat 是一个运行在 Claude Code、Cursor、Windsurf 等 AI 编程工具里的本地 skill。它通过结构化问卷理解用户的认知风格，生成可持久化、可切换、可复盘的 AI 协作 profile。

核心模型：

```text
Profile = AI 人设 Persona + 协作 Workflow + 项目记忆 Project Memory
```

---

## 1. 摘要

最好的协作，不只是对方能力强，而是对方刚好懂你的节奏。

和性格契合的人一起工作，会更高效，也更有火花。那 AI 为什么不能也这样？

Sorting Hat 解决的是这个被忽视的问题：不同用户需要完全不同的 AI 协作方式，但大多数 AI 默认用同一种语气、同一种结构、同一种反馈方式回答所有人。

Sorting Hat 通过问卷把用户答案转化为心理特征，再转化为 AI 行为规则，生成一份可写入本地工具配置的 AI 人设。同时，它是一个完整的多 profile 管理系统：用户可以为不同任务创建多套 AI 协作套装，为不同项目绑定独立的上下文记忆，并把"人如何主导 AI"的经验沉淀为可复用的 workflow。

## 2. 背景

### a. 项目故事与灵感来源

这个项目来自两个观察的交汇。

**第一，Anthropic 关于 AI 情绪表征的研究。** 2026 年，Anthropic 发表了论文《Emotion Concepts and their Function in a Large Language Model》，核心发现：Claude 内部存在 171 种可测量的情绪表征，真实影响 AI 输出行为——从回答详细程度，到面对压力时是坚持还是退让，再到反馈时是温柔还是直接，都受这些内部状态影响。更重要的是，论文发现 AI 的"温暖感"和"直率程度"之间存在内在冲突——越温暖越容易讨好用户，越直率越可能让用户不舒服。这不是靠一句系统提示能解决的问题，而是需要精确调校的参数。

这意味着：**AI 的"人格"不是比喻，是一组可以被测量、被设计、被精确配置的参数。**

**第二，一个反复出现的困惑。** 同样的 AI 工具，不同的人用出来的体验差异极大：有人嫌啰嗦，有人嫌太短；有人觉得反馈太直接，有人觉得正好。这些差异来自每个人不同的**认知风格**——有人思维发散（Ne 主导），有人收敛（Ni 主导）；有人决策时先看风险，有人先看机会；有人需要情感缓冲才能接受批评，有人认为直接说才是尊重。

**用错误的方式协作，不是 AI 不好用，是人设没配对。**

两件事叠在一起，产生了 Sorting Hat 的构想：像哈利波特里的分院帽一样，通过一次测试，把你分到最适合你的 AI 协作方式里。

### b. 为什么是现在

Claude Code 的 memory 系统解决了"人设持久化"的问题——写入 `~/.claude/memory/persona.md`，此后每次打开工具人设自动加载，无需重复设置。三件事叠加形成当下的机会窗口：

1. **Anthropic 情绪论文**：AI 人格是可精确调校的参数，不是玄学
2. **本地 memory 系统**：人设可以持久化，不需要每次重设
3. **多 agent 工具普及**：用户在不同项目中高频使用多个 AI 工具，需要跨项目的上下文管理

## 3. 目标

### a. 产品目标

帮助用户完成三件事：

1. 找到适合自己的 AI 人设，写入本地工具配置，一次配置长期生效。
2. 管理多套可切换的 AI 协作 profile，在不同任务场景下使用最合适的那一套。
3. 为每个项目建立独立的上下文记忆，让 AI 进入任意项目时自动了解背景和约定。

### b. 为什么重要

AI 越强，越需要被正确使用。真正影响体验的，不只是模型能力，而是 AI 是否知道：应该直接给结论还是带用户推导？应该温和反馈还是直接指出问题？应该如何在这个项目的技术约束下工作？

Sorting Hat 的价值，是把这些隐性的协作偏好和项目上下文，变成可保存、可复用、可迭代的本地资产。

## 4. 目标用户

### a. 主要用户

- 高频使用 Claude Code、Cursor、Windsurf 等 AI 编程工具的人
- 希望 AI 更懂自己工作方式的开发者、产品经理、设计师、研究者
- 经常需要在不同项目之间切换、希望 AI 自动适配项目上下文的深度用户
- 想把个人 AI 使用经验沉淀成 workflow 的用户

### b. 非目标用户

- 只想偶尔问 AI 一个简单问题的轻度用户
- 不希望配置任何本地文件或工具规则的用户
- 需要 SaaS 云端账号体系的场景

## 5. 价值主张

### a. 用户要完成的工作（Jobs to Be Done）

当我长期使用 AI 工作时，我希望它能理解我的认知风格、工作偏好和项目上下文，这样我不用每次重新解释自己，也不用反复纠正它。

### b. 解决了哪些痛点

- AI 太啰嗦 / 太简短
- AI 反馈太软 / 太冲
- AI 不知道用户什么时候想参与推导
- AI 不知道项目背景和协作约定
- 换了项目就要重新调教 AI
- 不同任务需要不同协作方式，但只有一套 AI 配置

### c. 它比现有方案好在哪里

Sorting Hat 不是给用户一段通用 prompt，而是提供一套完整流程：

```text
问卷识别 → 心理转化 → 人设生成 → 工具写入 → profile 管理 → 项目记忆 → workflow 沉淀
```

既解决"AI 怎么和你这个人工作"，也解决"AI 怎么在这个项目里工作"。

## 6. 使用方法

### a. 安装

```bash
git clone https://github.com/yakultyum/sorting-hat.git sorting-hat && cd sorting-hat && scripts/install-skill.sh --force
```

脚本会自动检测常见本地 skills 目录，安装完整 skill 包。也可指定安装位置：

```bash
scripts/install-skill.sh --dest ~/.cursor/skills --force
```

不能只复制 `SKILL.md`——网页问卷、本地服务器、规则文件都是 skill 的一部分。

### b. 创建 profile：问卷分院

```text
/sorting-hat
```

流程：

1. 选择简易版（11 题，约 5 分钟）或深度版（31 题，约 20 分钟）
2. 选择输出目标：Claude Code / Cursor / Windsurf / 通用 / 全部
3. 浏览器自动打开，在网页上完成问卷
4. 填写 profile 名称与适用场景
5. 点击保存，人设写入目标工具配置，同时保存到全局 profile 仓库

### c. 创建 profile：直接手写

不想做问卷？在网页欢迎页点击「直接手写人设」：

1. 填写 profile 名称
2. 多选适用场景
3. 直接写人设文本（AI 行为准则）
4. 保存，立即创建 profile

### d. 档案管理 UI

启动服务器后，点击「我的档案」进入档案管理页：

**查看所有 profile**：名称、版本（简易 / 深度 / 手写）、适用场景数，折叠卡片列表。

**展开 profile 可以：**
- 查看并编辑人设描述全文（`prompt.md`）
- 查看并逐段编辑 workflow（触发场景 / 主导动作 / AI 响应 / 反例 / 验证 prompt）
- 一键激活到 Claude Code，或输入项目路径激活到 Cursor / Windsurf

**档案页底部「项目记忆」全局区块**（所有 profile 共享）：
- 列出系统里所有已创建的项目记忆
- 点「+ 新建」直接创建，无需关联到某个 profile
- 每条项目记忆可展开查看和编辑（技术栈、约定、Gotcha、人设覆盖）

### e. 两层记忆架构

```
~/.claude/memory/persona.md          ← 全局基础人设
                                        末尾包含项目记忆读取指令

<project>/.sorting-hat/project.md   ← 项目记忆（所有 profile 共享）
                                        技术栈 + 约定 + Gotcha + 人设覆盖 + 协作规则
```

AI agent 启动时加载全局 persona，处理项目任务前主动检查并读取项目记忆。同一套基础人设，在不同项目里自动适配不同上下文。**这套设计不绑定 Claude Code，适用于所有支持文件读取的 AI agent。**

项目记忆与 profile 的关系：**一个项目只有一份 `project.md`，所有 profile 进入该项目都读同一份**，不按 profile 隔离。在档案页「项目记忆 → 新建」，填写：
- 项目路径
- 技术栈
- 约定（命名规范、提交规范等）
- Gotcha（容易踩的坑、特殊依赖）
- 人设覆盖（在这个项目里覆盖全局人设的部分内容）

### f. 沉淀 workflow

```text
/sorting-hat reflect
```

`reflect` 不保存聊天原文，只保存抽象协作规则：触发场景、人的主导动作、AI 应如何响应、反例、验证 prompt。生成后可在档案页按字段查看和编辑。

### g. 项目初始化

```text
/sorting-hat init
```

通过终端对话收集项目用途、技术栈、常用命令、命名规范和 gotcha，生成项目级 AI 配置 CLAUDE.md。

## 7. 解决方案

### a. 用户旅程

#### 首次创建 profile（问卷路径）

```text
/sorting-hat → 选择版本和输出目标
  ↓
浏览器打开 → 完成问卷（含交互偏好：喜欢什么 / 不喜欢什么）
  ↓
填写 profile 名称和适用场景
  ↓
保存 → 写入工具配置 + 全局 profile 仓库
```

#### 首次创建 profile（手写路径）

```text
网页欢迎页 → 直接手写人设
  ↓
填写名称、场景、人设文本 → 保存
  ↓
档案页 → 激活到目标工具
```

#### 为项目添加记忆

```text
档案页 → 底部「项目记忆」区块 → + 新建
  ↓
填写项目路径、技术栈、约定、Gotcha、人设覆盖
  ↓
写入 <project>/.sorting-hat/project.md
  ↓
所有 profile 进入该项目时均自动读取，覆盖全局人设对应准则
```

也可通过 `/sorting-hat init` 在终端完成——生成 CLAUDE.md 的同时自动写入 `.sorting-hat/project.md`，两个文件共用一次问答。

#### 管理和更新人设

```text
档案页 → 展开 profile
  ↓
直接在人设描述区编辑文本 → 点「完成」保存到 prompt.md
```

#### 沉淀协作经验

```text
/sorting-hat reflect → 选择 profile → 输入复盘内容
  ↓
生成 workflow.md → 档案页查看和编辑
```

### b. 核心功能

#### i. 两种 profile 创建路径

**问卷路径**：融合七个心理学框架的结构化问卷，简易版 11 题（约 5 分钟），深度版 31 题（约 20 分钟）。问卷最后新增「交互偏好」题——用户可以用自然语言描述喜欢什么（"偶尔用语气词"）和不喜欢什么（"不要在结尾总结"），直接写入人设。

**手写路径**：跳过问卷，直接填写人设文本。适合已经知道自己想要什么的用户，或快速为特定项目创建专用 profile。

#### ii. 七层心理学框架（问卷路径）

问卷不是凭感觉设计的选项，每道题背后都有对应的心理学理论，每个答案都有精确的三层转化规则：

```
用户答案 → 心理特征 → 人设语言
```

| 理论框架 | 捕捉的维度 |
|---|---|
| Anthropic 情绪研究 | AI 的情绪底色、温暖与诚实的平衡 |
| 荣格八维认知功能 | 推理深度偏好、反馈方式、问题解法风格 |
| OCEAN 大五人格 | 信息完整性偏好、风险敏感度 |
| 依恋理论 | 记忆连续性需求、错误容忍度 |
| 认知负荷理论 | 回复密度、格式偏好、专业假设深度 |
| 调节焦点理论 | 决策分析中机会与风险的呈现顺序 |
| 认知需求 + 控制点 | 推导参与深度、执行授权程度 |

#### iii. 档案管理 UI

网页端的全功能档案管理页，无需命令行操作：

- **profile 列表**：折叠卡片，标题行展示名称、默认标记、版本、场景数
- **人设编辑**：点「编辑」切换为 textarea，直接改写人设文本，点「完成」保存到 `prompt.md`
- **workflow 编辑**：每条 workflow 的五个字段（触发场景、主导动作、AI 响应、反例、验证 prompt）各自独立编辑
- **激活**：一键激活到 Claude Code（全局），或输入项目路径激活到 Cursor / Windsurf，保存绑定记录
- **项目记忆管理**：为绑定项目新建 / 查看 `.sorting-hat/project.md`

#### iv. 两层记忆架构

```
~/.claude/memory/persona.md          ← 全局基础人设（通用行为准则）
                                        末尾固定包含：
                                        "每次开始任务前检查 .sorting-hat/project.md
                                         如存在则读取并覆盖对应准则"

<project>/.sorting-hat/project.md   ← 项目记忆
                                        技术栈、约定、Gotcha、人设覆盖
```

设计原则：项目记忆文件独立于 Claude Code，适用于所有 AI agent。AI 通过 persona.md 里的显式指令主动读取，而不是被动加载。

#### v. 人设文件持久化

支持多目标输出：

- **Claude Code**：`~/.claude/memory/persona.md`（全局，每次对话自动加载）
- **Cursor**：`.cursor/rules/sorting-hat.mdc`（项目级）
- **Windsurf**：`.windsurfrules`（项目级）
- **Generic**：浏览器 / 终端复制纯 prompt（通用）
- **All**：同时生成所有格式

写入已有文件前自动生成 `.bak` 备份，并保存原始答题数据 JSON。

#### vi. Workflow Sorting

沉淀"人如何主导 AI"的抽象协作规则，不保存完整聊天原文。每条 workflow 包含：

- 触发场景
- 人的主导动作
- AI 应如何响应
- 反例
- 验证 prompt

在档案页可按字段逐段查看和编辑；通过 `/sorting-hat reflect` 从复盘文本自动生成。

### c. 技术实现

- **类型**：Markdown skill + Python 本地服务器 + HTML 网页（单文件 SPA）
- **前端**：`web/index.html`，负责问卷、档案管理 UI、保存请求
- **后端**：`web/server.py`，Python HTTP server，端口 7432，负责：
  - 页面托管（`GET /`）
  - 已有答案读取（`GET /state`）
  - profile 列表（`GET /profiles`，含 prompt、workflow 字段解析）
  - 新问卷保存（`POST /save`，写工具文件 + profile 仓库）
  - 手写创建 profile（`POST /profiles`）
  - 激活 profile（`POST /profiles/<id>/activate`）
  - 编辑人设文本（`PUT /profiles/<id>/prompt`）
  - 编辑 workflow 字段（`PUT /profiles/<id>/workflows/<wid>`）
  - 创建项目记忆（`POST /projects`）
  - 读取所有项目记忆（`GET /projects/all`，跨 profile 全局视图）
  - 读取单个项目记忆（`GET /projects?dir=<path>`）
- **存储**：全部本地，不上传服务器
- **测试**：单元测试覆盖目标解析、备份、profile store、profiles/use/reflect
- **发布检查**：脚本校验资源完整、web 与 skill/web 同步、Python 可编译、JSON 有效

### b. 未来规划

**① AI 搭子形象 × 桌面桌宠**

当前 Sorting Hat 定义的是 AI 的行为准则，但没有给它一个「形象」。未来计划结合桌面桌宠（如 Live2D / 轻量桌面 App），把 profile 里的情绪底色、语气风格映射成一个有表情、有动作、有个性的视觉角色——它不只是在对话框里用文字回答你，而是一个真实坐在你桌面角落的 AI 搭子。不同 profile 对应不同角色形象：严格工程搭档是沉默专注的那种，Darwin 优化器可能带点挑剔的表情，写作顾问或许更有温度。分院的结果不只写进配置文件，也塑造了你和 AI 之间关系的「样子」。

**② Profile 市场：分享和订阅协作套装**

高质量的 AI 协作方式值得被传播。未来允许用户把自己的 profile（去掉个人信息后）发布到 Profile 市场，其他人可以一键订阅「资深代码审查员」「产品方案评审官」「严苛写作编辑」等经过验证的协作套装，而不是每次从零开始答问卷。团队也可以维护私有 profile 库，统一团队 AI 协作标准。

**③ 人设漂移检测**

人的偏好会随时间和场景改变，但 profile 是静态的。未来通过分析用户对 AI 回答的隐性反馈（频繁追问 = 信息密度不够、频繁打断 = 推理太慢等），在用户行为与当前 profile 明显不匹配时，主动提示「你最近的使用习惯和当前人设有偏差，要调整吗？」——让 profile 随人成长，而不是一次配置永远不变。

**④ 团队级协作套装**

同一个项目里，不同角色需要 AI 以不同方式工作：技术 lead 需要 AI 保持严格、不妥协；新人需要 AI 耐心解释基础；PM 需要 AI 快速给选项而不是展开推导。未来支持团队维护一套 profile 矩阵，按角色自动匹配，让 AI 知道「现在在和谁说话」，而不是对所有人用同一套准则。

**⑤ 从历史对话自动蒸馏 Workflow**

`/sorting-hat reflect` 目前需要用户主动输入复盘内容。未来探索从历史对话中自动识别「人纠正了 AI」「人重新说明了要求」「某种交互模式反复出现」等信号，自动提炼成候选 workflow 规则，让用户确认后沉淀——从「人工总结」走向「自动涌现」。

**⑥ `/sorting-hat routes`：按任务类型自动切换 Profile**

一个人在一天里可能同时扮演多个角色：写代码时需要严格工程搭档，写文档时需要写作编辑顾问，做决策分析时需要平衡风险和机会的顾问。未来支持为不同任务类型（文件类型、关键词、时间段）绑定不同 profile，AI 自动感知当前任务性质，切换到最合适的协作模式。

### c. 明确不做的事

- 不做独立 SaaS，本地优先
- 不上传用户答案或聊天内容
- 不把心理标签当作结论，只作为生成 AI 行为规则的中间层
- 不保存完整聊天原文，只保存抽象 workflow

## 一句话总结

Sorting Hat 不只是帮你找到适合你的 AI，更帮你管理多套 AI 协作方式、为每个项目建立独立记忆，并把你如何主导 AI 做好项目的经验沉淀下来，跨项目复用。
