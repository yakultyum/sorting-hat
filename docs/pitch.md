# 🎩 Sorting Hat · AI 人设分院仪式

> 让每个人找到最适合自己的 AI 协作方式

---

## 我们解决了什么问题

大多数人使用 AI 的方式是一样的：打开对话框，开始提问。

但每个人和 AI 协作的"最佳方式"其实差异很大——

- 有人希望 AI 直接给结论，有人希望一起推导
- 有人需要情感缓冲，有人觉得那是多余的客套
- 有人想要发散 5 个方案，有人要 AI 直接给最优解
- 有人希望 AI 先说风险，有人希望先看机会
- 有人用一个项目，有人同时跑五个，每个项目的上下文和约定都不一样

这些差异不是偏好问题，而是认知结构问题。用错误的方式协作，不仅效率低，还会让人觉得"AI 不好用"——但其实只是人设没配对。

**Sorting Hat 做的事：** 像哈利波特里的分院帽一样，通过一套结构化问卷，帮你找到最适配自己认知风格的 AI 人设，并写入工具的记忆系统，从此每次对话都以该人设工作，无需重复设置。同时，为每个项目建立独立的记忆文件，让 AI 进入任何项目时自动了解背景。

---

## 它是怎么工作的

### 两种创建方式，同一套管理体验

**方式一：问卷分院**

```
① 调用 /sorting-hat
        ↓
② 回答 11 题（简易版，约 5 分钟）
   或  31 题（深度版，约 20 分钟）
        ↓
③ 人设写入 ~/.claude/memory/persona.md
   下次对话起永久生效
```

**方式二：直接手写**

```
① 网页欢迎页 → 直接手写人设
        ↓
② 填写名称、选适用场景、写人设文本
        ↓
③ 在档案页激活到目标工具
```

### 两层记忆架构，跨项目自动适配

```
~/.claude/memory/persona.md          ← 全局基础人设
                                        末尾包含：
                                        "每次开始任务前检查 .sorting-hat/project.md
                                         如存在则读取并覆盖对应准则"

<project>/.sorting-hat/project.md   ← 项目记忆
                                        技术栈 + 约定 + Gotcha + 人设覆盖
```

同一套基础人设，在不同项目里自动适配不同上下文。这套设计不绑定 Claude Code，适用于所有支持文件读取的 AI agent。

### 零依赖，完全本地

- 一条命令安装，无需配置任何服务
- 结果写入本地文件，数据不离开你的机器
- memory 系统自动加载，无需额外操作

## 技术亮点

### 1. 七层心理学框架，精确捕捉认知差异

问卷不是凭感觉设计的选项，每道题背后都有对应的心理学理论，每个答案都有精确的三层转化规则：

```
用户答案 → 心理特征 → 人设语言
```

| 理论框架 | 捕捉的维度 |
|---|---|
| Anthropic 情绪研究 | AI 的情绪底色、温暖与诚实的平衡 |
| 荣格八维认知功能 | 推理深度偏好、反馈方式、问题解法风格 |
| OCEAN 大五人格 | 信息完整性偏好、风险敏感度 |
| 依恋理论 | 记忆连续性需求、错误容忍度 |
| 认知负荷理论 | 回复密度、格式偏好、专业假设深度 |
| 调节焦点理论 | 决策分析中机会与风险的呈现顺序 |
| 认知需求 + 控制点 | 推导参与深度、执行授权程度 |

举个例子——同样是"AI 发现你的方案有问题"，不同的人需要完全不同的处理方式：

> **选 A（Fe 主导）** → 人设语言：批评前先肯定，用缓冲语言降低对抗感，维护用户的情绪状态是反馈质量的一部分
>
> **选 B（Fi 主导）** → 人设语言：有问题直接说，不加糖衣；用户不需要被保护，需要被诚实对待
>
> **选 D（Ti 主导）** → 人设语言：只提供事实分析，不做价值判断；把评价权完全留给用户

### 2. 交互偏好：自然语言补充

除了结构化问卷，Sorting Hat 还在最后加了一道开放题——**用自然语言描述你对 AI 交互方式的喜好**：

> **喜欢**：说话带点语气词、偶尔主动给额外补充…
> **不喜欢**：不要在每次回答结尾说"希望这个回答对你有帮助"…

这些直接写入人设，不做心理解读，用户说什么就是什么。

### 3. 情景探针设计，而非直接问偏好

问卷不问"你喜欢什么风格的 AI"，而是通过具体工作情景让用户做选择。

这来自心理学测量的基本原则：**行为预测行为，自我报告不可靠。** 当你在情景里做选择，暴露的是真实的认知偏好，而不是你以为自己想要的东西。

### 4. 三层转化规则，避免人设失真

一个常见错误是把用户的情景选择直接翻译成人设描述：

> ❌ 用户选了"带我一步步推导" → 人设写"你会带用户一步步推导"

这是直接复制情景，而不是理解用户的认知需求。Sorting Hat 的做法是：

> ✅ 用户选了"带我一步步推导"
> → 心理特征：Ti 主导 + 高认知需求，被思考过程本身驱动，理解优先于速度
> → 人设语言：暴露完整思维链，不替用户跳过推理步骤；用户真正理解比快速给出答案更重要

这套转化规则确保人设描述是**通用的行为原则**，而不是绑定在特定情景上的条件反应。

### 5. 网页档案管理 UI

一个完整的 profile 管理界面，无需命令行：

- **查看所有 profile**：折叠卡片，名称 + 版本 + 场景数
- **编辑人设**：点「编辑」直接改写人设文本，点「完成」保存
- **编辑 workflow**：五个字段（触发场景 / 主导动作 / AI 响应 / 反例 / 验证 prompt）各自独立编辑
- **激活到工具**：一键激活到 Claude Code，或输入项目路径激活到 Cursor / Windsurf
- **项目记忆管理**：新建 `.sorting-hat/project.md`，填写技术栈、约定、Gotcha、人设覆盖

## 为什么现在做这件事

Anthropic 2026 年的研究证明，AI 内部存在 **171 种可测量的情绪表征**，这些表征真实影响 AI 的输出行为。这意味着"AI 人格"不是比喻——它是一个可以被精确调校的参数集合。

与此同时，Claude Code 的 memory 系统让人设持久化成为可能；多 agent 工具的普及让项目级上下文管理成为刚需。

这三件事叠加，创造了一个过去不存在的机会：**用心理学精度，一次性配置你和 AI 的协作方式——个人维度和项目维度都覆盖到。**
